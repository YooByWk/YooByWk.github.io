# MongoDB

`MongoDB` 대표적인 `NoSQL`로 도큐먼트 기반의 데이터베이스.

전통적인 관계형 데이터베이스(RDB)가 행과 열을 사용하는 것과 다르게 JSON과 유사한 BJON(Binary JSON) 형식의 문서를 활용하여 데이터를 저장한다.

## 특징

### 1. 문서 모델

데이터가 하나의 문서에 캡슐화 되어 저장된다. 이 문서는 중첩된 구조를 가질 수 있다.

RDB의 Row 에 해당하며, 데이터의 지역성이 높아 관련 정보 접근이 빠르다.

### 2. 유연한 스키마

컬렉션 내 Document 들은 서로 다른 필드를 가질 수 있다.

이러한 특징으로 인해 서비스 변경/확장에 따른 스키마 변경 비용이 낮다.

### 3. 고가용성

Replica Set 을 통해 자동 장애 복구 (Failover) 및 데이터 다중화를 지원.

만약 프라이머리 서버에 장애가 발생시, 세컨더리 서버를 리더 서버로 사용(자동)

### 4. 수평 확장성

샤딩을 통해 대용량 데이터를 여러 서버에 분산 저장할 수 있음.

정확히 말하면 자동적으로 진행해주어 `auto-sharding` 이라 볼 수 있음

### RDB와 비교

|  구분  |                            MongoDB (NoSQL)                             |                   RDB (SQL)                    |
| :----: | :--------------------------------------------------------------------: | :--------------------------------------------: |
| 스키마 |                          유연함 (Schemaless)                           |             고정적 (Fixed Schema)              |
|  확장  |                       수평 확장 (Sharding) 용이                        |           수직 확장 (Scale-Up) 중심            |
|  관계  | 데이터 중복을 허용하고 임베딩(Embedding) 또는 참조(Referencing)로 표현 |          정규화를 통한 Join으로 표현           |
| 무결성 |                애플리케이션 레벨에서 보장 (Mongoose 등)                | 데이터베이스 레벨에서 보장 (트랜잭션 제약조건) |

## 사용

- 콘텐츠 관리 및 블로그 등 : 정형화 되지 않은 다양한 데이터의 메타데이터와 같은 정보를 유연하게 저장하는데 사용
- 수정이 빈번한 요소 : 프로필, 카탈로그 등 수정이 많은 데이터에 사용
- 로그 및 이벤트 데이터 : "비정형"로그 데이터를 고속 저장 & 인덱싱에 유리함
